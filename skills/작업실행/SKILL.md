# 작업실행

사용자가 요청한 작업을 판단 → (필요 시) 계획 → 실행하는 워크플로우입니다.

---

## Step 1. 복잡도 판단

요청을 분석하여 **간단한 작업인지 복잡한 작업인지** 먼저 판단합니다.

### 모호한 요청

요청이 모호하거나 범위가 불분명한 경우, **바로 실행하지 않고** 사용자에게 구체적으로 무엇을 할지 확인합니다.

예시:
- "이거 좀 고쳐줘" → 무엇을 어떻게 고칠지 확인
- "성능 개선해줘" → 어떤 부분의 성능인지, 기대하는 수준이 있는지 확인
- "리팩토링해줘" → 어떤 방향으로, 어떤 범위인지 확인

### 간단한 작업

다음 조건에 해당하면 간단한 작업입니다:
- 수정할 파일이 1~3개
- 변경 범위가 명확하고 조사가 필요 없음
- 한 가지 기능/수정에 집중

→ **Step 2를 건너뛰고 바로 실행합니다.** 계획 보고 없이 작업 후 결과만 보고합니다.

### 복잡한 작업

다음 조건 중 하나라도 해당하면 복잡한 작업입니다:
- 수정할 파일이 4개 이상
- 여러 영역에 걸친 변경 (프론트+백엔드+DB 등)
- 사전 조사/분석이 필요
- 아키텍처 결정이 포함됨

→ **Step 2에서 계획을 수립합니다.**

---

## Step 2. 계획 수립 (복잡한 작업만)

### 2a. subtask 분해

1. 전체 작업을 subtask로 분해
2. 각 subtask 간 의존 관계 판별
3. 필요 시 관련 파일/코드 사전 조사

### 2b. 실행 방식 결정

subtask의 성격을 보고 **순차 실행**인지 **팀 병렬 실행**인지 결정합니다.

#### 순차 실행 (혼자)

다음 경우에 순차로 직접 실행합니다:
- subtask 간 의존성이 강함 (앞 결과가 뒤에 필요)
- 동일 파일을 여러 단계에서 수정
- 코드 수정이 대부분이고 조사/검증 비중이 적음

#### 팀 병렬 실행

다음 경우에 팀원을 만들어 병렬로 실행합니다:
- 독립적인 subtask가 2개 이상 동시 진행 가능
- 웹 검색, 코드 리뷰, 최적화 분석 등 **조사 작업**이 구현과 동시에 가능
- 서로 다른 영역의 코드 수정이 **파일 겹침 없이** 분리 가능

### 2c. 팀 구성 계획 (병렬 실행 시)

팀원을 만들 경우 다음을 계획합니다:

```
## 팀 구성

| 팀원 | 역할 | 담당 subtask | 수정/조사 범위 |
|------|------|-------------|---------------|
| 조사 담당 | 웹 검색 + 문서 분석 | subtask A | 읽기 전용 |
| 구현 담당 A | 프론트엔드 구현 | subtask B | apps/web/src/app/feature/ |
| 구현 담당 B | API 라우트 구현 | subtask C | apps/web/src/app/api/feature/ |
| 검증 담당 | 빌드 + 테스트 | subtask D | 읽기 전용 |
```

**팀 구성 원칙:**
- 작업 규모에 따라 2~4명 유동적 배정
- **코드 수정 팀원 간 파일이 절대 겹치지 않도록** 범위를 명확히 지정
- 조사/검증 역할은 읽기 전용으로 명시
- 팀원 생성 방식은 **상황에 따라 선택** (아래 Step 3 참고)

### 2d. 계획 보고

계획을 사용자에게 보고하고 **승인 후 실행**합니다.

```
## 작업 계획

### 개요
- 작업: [설명]
- 방식: 순차 / 팀 병렬
- subtask: N개

### 실행 순서
1. subtask A — 설명
2. subtask B + C — 동시 진행 (팀원 배정)
3. subtask D — B, C 완료 후

### 팀 구성 (병렬 시)
| 팀원 | 역할 | 범위 |
|------|------|------|
| ... | ... | ... |

→ 진행할까요?
```

---

## Step 3. 실행

### 순차 실행

subtask를 하나씩 직접 완료합니다.

### 팀 병렬 실행

#### 방식 선택

| 방식 | 언제 사용 | 도구 |
|------|----------|------|
| **subagent** | 독립적 작업을 동시 실행, 팀원 간 소통 불필요 | Task 도구 (`subagent_type: "general-purpose"`) |
| **팀(Team)** | 팀원 간 실시간 소통이 필요하거나, 장시간 협업이 필요한 경우 | TeamCreate → Task (팀원 생성) → SendMessage (소통) |

대부분의 경우 **subagent로 충분**합니다. 팀원끼리 중간 결과를 주고받아야 하는 경우에만 Team을 사용합니다.

#### 실행

1. 선택한 방식으로 팀원을 **동시에** 생성
2. 각 팀원에게 전달할 컨텍스트:

```
프로젝트: [CLAUDE.md의 프로젝트명]
기술 스택: [CLAUDE.md의 기술 스택]
프로젝트 루트: [현재 git root]
패키지 매니저: [CLAUDE.md 참고]
담당 작업: [구체적 작업 내용]
수정 대상 파일: [파일 목록] (조사/검증 팀원은 "읽기 전용 — 코드 수정하지 말 것")
주의사항: [다른 팀원과 겹치지 않는 파일 범위]
```

3. 모든 팀원 완료 후 결과를 취합

### 실패 대응

| 상황 | 대응 |
|------|------|
| 팀원 작업 실패 | 결과 분석 → 직접 처리 |
| 파일 충돌 | 순차 모드로 전환하여 직접 병합 |
| 빌드 에러 | 즉시 수정 후 재빌드 |
| 방향 틀어짐 | 작업 중단 → 사용자에게 보고 → 재수립 |

---

## Step 4. 검증

dev 서버가 켜져 있으면 **dev 서버에서 에러 없이 동작하는지** 확인합니다 (브라우저, Chrome DevTools MCP 등 활용).

**verify 스킬은 다음 경우에만 실행:**
- 보안 관련 변경 (인증, RLS, API 키 등)
- 사용자가 명시적으로 요청한 경우

---

## Step 5. 완료 보고

```
## 작업 완료

| subtask | 상태 | 변경 파일 |
|---------|------|----------|
| A | 완료 | file1.ts, file2.ts |
| B | 완료 | file3.ts |

### 요약
- [한 줄 요약]
```
